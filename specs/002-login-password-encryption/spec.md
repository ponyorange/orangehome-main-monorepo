**Feature Branch**: `002-login-password-encryption`  
**Created**: 2026-03-30  
**Status**: Draft  
**Input**: User description: "@web_platform @web_builder 这两个项目登录的时候，把密码加密一下；然后在 @server_bside 这个项目收到加密后的密码后先进行解密，再继续后续的登录操作。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 平台端登录且密码不以明文随请求发送 (Priority: P1)

用户在 **Web 平台**（`web_platform`）使用账号与密码登录时，提交的密码在到达服务端用于校验之前，在网络上传输的内容中不得以可读的明文密码形式出现；校验逻辑与登录成功后的体验（会话、跳转等）与现有行为一致。

**Why this priority**: 平台是主入口之一，且本需求首先覆盖该平台登录链路。

**Independent Test**: 仅在平台端开启保护后，用有效账号走完整登录流程，确认可登录成功；同时用抓包或等价方式确认登录请求负载中不包含用户输入的明文密码。

**Acceptance Scenarios**:

1. **Given** 用户持有有效账号与密码且未登录，**When** 在平台登录页提交登录，**Then** 登录成功并进入与现有产品一致的已登录状态。
2. **Given** 用户输入错误密码，**When** 提交登录，**Then** 登录失败且提示与现有安全策略一致（不泄露账号是否存在等敏感信息）。
3. **Given** 用户正在登录，**When** 观察登录请求中的密码相关字段，**Then** 该内容不可直接识读为明文密码（与「仅 HTTPS 传输明文表单」相比有额外保护）。

---

### User Story 2 - 搭建器端登录且密码不以明文随请求发送 (Priority: P2)

用户在 **Web 搭建器**（`web_builder`）使用账号与密码登录时，同样不得在请求中以明文密码形式发送；登录成功后的行为与现有一致。

**Why this priority**: 与平台并列的第二前端，需一致的安全基线。

**Independent Test**: 仅在搭建器端验证：有效凭证可登录；请求中密码相关负载不可直接识读为明文。

**Acceptance Scenarios**:

1. **Given** 用户持有有效账号与密码，**When** 在搭建器登录流程中提交登录，**Then** 登录成功且会话行为与现有一致。
2. **Given** 用户提交错误密码，**When** 完成登录请求，**Then** 登录失败且错误提示符合现有约定。

---

### User Story 3 - 服务端在校验前恢复凭证并拒绝异常负载 (Priority: P1)

**B 端服务**（`server_bside`）在收到登录请求后，必须先将受保护的密码内容恢复为可用于既有登录校验流程的形式，再执行原有登录逻辑；若负载被篡改、过期或无法解密，则不得当作有效密码通过校验，并返回与现有错误处理一致的安全表现。

**Why this priority**: 无服务端恢复与衔接，前端保护无法落地；与 US1 共同构成最小端到端价值。

**Independent Test**: 用自动化或手工发送合法保护负载可登录；发送损坏或伪造负载时不能登录且无不恰当信息泄露。

**Acceptance Scenarios**:

1. **Given** 请求携带与协议一致且有效的受保护密码负载，**When** B 端服务处理登录，**Then** 先完成恢复再执行既有校验，成功时行为与改造前一致。
2. **Given** 请求携带无法解密或校验失败的负载，**When** B 端处理登录，**Then** 拒绝认证，且不暴露内部实现细节（如具体密码、密钥材料）。

---

### Edge Cases

- 客户端与服务端**保护协议版本不一致**时：用户应收到可理解的失败提示，并可重试；不得静默成功或崩溃。
- **时钟偏差**（若保护方案依赖时间窗口）：在合理偏差范围内仍应能登录；超出范围时失败提示应友好。
- **重放**：同一受保护负载被重复提交时，行为应符合产品既有安全策略（若当前无防重放，则至少不因本功能引入新的明文暴露）。
- **仅 HTTPS、无额外保护时的降级**：不得在无约定的情况下回退为明文密码随请求发送；若必须降级，须有明确产品决策与风险说明（默认不采用明文回退）。
- **HTTP 公网 IP（非安全上下文）**：浏览器不提供 `crypto.subtle` 时，实现上采用与 BFF 一致的 **纯 JS 加密降级**（`node-forge`），使负载仍可密文传输；**不**将此类场景等同于「可明文登录」，除非显式开启开发开关（见 `quickstart.md`）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `web_platform` 与 `web_builder` 的密码登录路径 MUST 在提交时以**受保护形式**传输密码，不得以可读的明文密码作为请求中的直接载荷（在既有传输加密之外仍满足此条）。
- **FR-002**: `server_bside` MUST 在调用既有「用户名/密码校验」逻辑之前，将受保护内容**恢复**为校验步骤可使用的形式；恢复失败 MUST 视为认证失败。
- **FR-003**: 改造后 MUST 保持与改造前等效的登录成功/失败业务结果（除安全增强所要求的必要差异外），包括会话建立、错误提示类别、审计需求（若有）。
- **FR-004**: 系统 MUST 对密钥与解密所需敏感材料实施保护（仅存于安全边界内，不得写入前端可分发资源或公开仓库）。
- **FR-005**: 若存在多环境（开发/预发/生产），各环境的保护参数 MUST 可配置且互不混用，避免跨环境重放或误用。

### Key Entities

- **受保护的登录负载**：表示用户密码经保护后的数据，随登录请求传递，可被 B 端服务验证并恢复。
- **登录会话结果**：与现有产品一致的成功或失败结果（含用户可见状态），不因本功能改变业务定义。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在标准网络条件下，使用有效凭证的用户在 **95%** 以上的登录尝试中，可在 **5 秒内** 从提交表单到达与改造前相同的「已登录」或明确失败状态（不含用户自身输错密码）。
- **SC-002**: 在可重复的抽检程序下，**100%** 抽检用例中，旁观者不能从登录过程中传递的凭据信息直接还原为用户输入的密码（抽检程序在发布前书面约定并归档）。
- **SC-003**: 恶意或损坏的受保护负载在测试中 **100%** 不能导致误认证为有效用户。
- **SC-004**: 上线后首周内，与登录相关的客服/支持工单中，因「无法登录」且归因于本改造的占比 **不超过 2%**（或团队定义的可接受阈值），否则须回滚或热修。

## Assumptions

- 现有登录已基于 **HTTPS**；本功能是在此之上对**密码字段**增加传输形态保护，而非替代 TLS。
- 用户仍使用**用户名/邮箱 + 密码**等与当前一致的凭据模型；不改变为第三方-only 登录。
- `server_bside` 已为登录提供或代理既有认证能力；本功能仅改变密码到达该服务时的形态及服务端恢复步骤。
- 算法与密钥分发细节在**规划/设计阶段**确定；规格层面只约束可验证的安全与行为结果。
- `web_platform` 与 `web_builder` 可共享同一套「保护协议」约定，以减少分叉；若因发布节奏需分阶段，须在计划中说明里程碑。
