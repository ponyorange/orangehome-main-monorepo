# Feature Specification: 编辑器「已保存」状态与预览前保存

**Feature Branch**: `010-preview-save-dirty-state`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: User description: 用一个state记录一下当前有没有保存过，编辑器初始化时默认是true，当编辑组件的时候，也就是schema有变动的时候，设置为false。等下次保存成功后设置为true。预览的时候，先判断有没有保存过，如果没有，就先保存，然后再预览，注意要有loading状态。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 编辑后知道「尚未保存到持久化」(Priority: P1)

作为页面搭建编辑者，我在编辑器中修改页面结构或样式（对当前可编辑文档产生实质变更）后，系统应能区分**当前编辑内容是否与最近一次成功持久化一致**；在刚打开已对齐后端的页面或完成一次成功保存后，我应处于「已保存」认知状态；一旦我改动文档，应进入「相对上次保存已有变更」的状态，以便后续流程（如预览）能据此决策。

**Why this priority**：这是整条的逻辑基础；没有可靠的「是否已保存」判断，预览前保存无法定义。

**Independent Test**：仅实现状态翻转（不含预览）时，可通过：打开页面 → 确认初始为「已保存」→ 任意合法编辑 → 变为「未保存」→ 执行保存并成功 → 回到「已保存」。

**Acceptance Scenarios**:

1. **Given** 编辑器已加载且与最近一次成功持久化内容一致，**When** 用户尚未做任何会改变可编辑文档的变更，**Then** 系统应将当前文档视为**已与持久化对齐**（对应实现中的「已保存」语义）。
2. **Given** 当前为已与持久化对齐状态，**When** 用户对可编辑文档做出会改变其序列化结果的变更（如增删改组件、修改属性或样式等），**Then** 系统应将状态切换为**相对上次成功保存已有未持久化变更**。
3. **Given** 当前存在未持久化变更，**When** 用户触发保存且持久化**成功完成**，**Then** 系统应回到**已与持久化对齐**状态。

---

### User Story 2 - 保存失败不改变「已对齐」误判 (Priority: P2)

作为编辑者，当我在有未保存变更时点击保存，若网络或服务错误导致保存失败，我不希望系统误认为「已经保存成功」；我应仍被视为有未持久化变更，并能看到或感知失败，以便重试或修正。

**Why this priority**：错误保存若把状态标成已保存，会导致预览展示旧版本而用户以为已更新，风险高。

**Independent Test**：模拟保存失败，确认「未持久化」状态保留，且不会出现「预览基于错误假设已保存」的行为。

**Acceptance Scenarios**:

1. **Given** 存在未持久化变更，**When** 保存操作结束且结果为**失败**，**Then** 系统**不得**将状态切换为已与持久化对齐。
2. **Given** 保存失败，**When** 用户修正问题后再次保存成功，**Then** 系统应切换为已与持久化对齐。

---

### User Story 3 - 预览前自动保存并呈现加载 (Priority: P1)

作为编辑者，当我点击**预览**时，若当前仍有相对上次保存的未持久化变更，我希望系统在打开预览**之前**先完成保存，并在等待保存（及随后进入预览）的过程中看到**明确的加载提示**，避免我误以为预览已反映最新改动；若当前已与持久化对齐，我应能直接进入预览而无需多余等待。

**Why this priority**：这是用户描述中的显式目标，且与 US1 强相关；与 US1 同列为 P1 因产品价值闭环在「预览见最新」。

**Independent Test**：在未保存状态下点预览 → 观察到加载态 → 保存成功后预览展示与当前编辑一致；在已保存状态下点预览 → 无因「补保存」产生的不必要长时间阻塞（允许极短过渡）。

**Acceptance Scenarios**:

1. **Given** 存在未持久化变更，**When** 用户发起预览，**Then** 系统应**先**完成持久化保存，**再**进入预览流程，且在整个必要等待期间展示**可感知的加载状态**。
2. **Given** 已与持久化对齐，**When** 用户发起预览，**Then** 系统应**不**因「补保存」强制插入完整保存流程（除非产品另有强制校验需求，见 Assumptions）。
3. **Given** 预览触发的保存失败，**When** 流程结束，**Then** 用户应得到失败反馈，且**不应**在误以为已保存成功的情况下展示「代表已持久化版本」的预览结果。

---

### Edge Cases

- **仅选中/仅画布平移/仅 UI 折叠**等不改变持久化文档的操作：不应错误地将状态标为未保存（与「schema / 可编辑文档变更」定义一致即可）。
- **撤销/重做**若会改变文档与持久化对齐关系：状态应随文档内容相对上次保存是否一致而更新（见 Assumptions）。
- **并发或多标签**：若产品允许多处编辑同一资源，以「最后一次成功保存」为基准的定义需在 Assumptions 中说明是否在本期范围内。
- **用户快速连点预览**：应合并或排队，避免重复提交保存或重复打开多个预览。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：系统 MUST 维护一种可判定的**文档相对持久化对齐状态**：在「与最近一次成功持久化一致」与「存在未持久化变更」之间区分。
- **FR-002**：在编辑器完成加载且当前展示内容与该次加载所依据的持久化版本一致时，状态 MUST 为**已与持久化对齐**（对应用户所述初始化默认「已保存」语义）。
- **FR-003**：每当可编辑文档发生会改变其持久化结果的变更时，状态 MUST 切换为**存在未持久化变更**，直至下一次保存成功。
- **FR-004**：保存操作**成功完成**后，状态 MUST 切换为**已与持久化对齐**；保存**未成功**则 MUST **不**切换为已对齐。
- **FR-005**：用户发起**预览**且当前为**存在未持久化变更**时，系统 MUST **先**完成保存，**再**进入预览，并在等待期间展示**加载状态**。
- **FR-006**：用户发起**预览**且当前**已与持久化对齐**时，系统 MUST **不**为满足本需求而强制执行一次完整保存（除非 Assumptions 中另有规定）。
- **FR-007**：预览路径上若保存失败，系统 MUST 向用户传达失败，且 MUST **不**呈现会误导用户「已按最新内容持久化并预览」的结果。

### Key Entities

- **文档对齐状态**：布尔或等价枚举（已与持久化对齐 / 存在未持久化变更）；由编辑会话持有，不替代服务端真源。
- **持久化保存**：将当前可编辑文档成功写入产品定义的权威存储（如项目页版本）；以**成功回调/响应**为成功判据。

## Orange Editor / web_builder *(conditional)*

- **ARCH-001**：实现须遵循 `.specify/memory/constitution.md`（schema 为真源、经批准的变更路径、store 边界、API 层约定）。
- **ARCH-002**：状态应与现有 `schema` 更新流、保存 API、预览入口解耦清晰，避免多处重复维护「是否脏」逻辑而不一致。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：在标准走查脚本（打开 → 编辑 → 保存 → 预览）下，**100%** 步骤中「对齐状态」与用户预期一致（编辑后未保存前为未对齐，保存成功后为已对齐）。
- **SC-002**：在未保存状态下触发预览的 **10** 次试验中，**至少 9** 次用户能在 **3 秒内**感知到加载提示（或等价阻塞 UI），且随后预览内容与保存前编辑区一致（在单用户、单会话、网络正常前提下）。
- **SC-003**：保存失败场景中，**零例**出现「状态显示已对齐但预览仍为旧持久化内容」的组合（抽样走查）。

## Assumptions

- 「**schema 有变动**」等同于**可编辑页面文档**相对上次成功持久化内容已变化；实现上可与「当前 schema 与上次成功保存快照不一致」同义，具体由技术方案定义快照边界（不含纯视图状态）。
- **初始化为已对齐**：指本次会话加载的文档与加载时所依据的持久化版本一致；若为新空白页且无未提交变更，视为已对齐。
- **预览**指产品当前提供的「从编辑态进入预览/运行时查看」的主路径；若存在多种预览入口，默认**全部**适用同一规则，除非后续规格单列例外。
- **撤销/重做**：若会改变文档内容，应参与脏标记（与「是否相对上次保存有差异」一致）。
- **加载状态**：至少包含可见的进行中指示（文案、遮罩、按钮 loading 等任选其组合），足以让用户理解「正在保存或正在打开预览，请勿重复操作」。
