# Feature Specification: 画布叶节点直出组件 DOM，编辑器装饰外置

**Feature Branch**: `008-schema-selection-portal`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: User description: 当前 `SelectableSchemaNode` 在真实组件（如按钮）外仍包有多层 `div`（含 `data-schema-id` 的外壳与内层 `contentRef` 等），与「真正渲染的组件」DOM 路径不一致。用户希望 **直接渲染组件真实内容**，**不再为编辑器目的增加包裹用的结构节点**；若必须在组件周围呈现选中框、悬停、拖动手柄等，应通过 **在组件子树之外挂载 UI** 的方式实现（例如通过 Portal 挂到画布根或专用层，具体实现由技术方案决定）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 叶节点在 DOM 中与真实组件对齐 (Priority: P1)

作为搭建编辑者或对接运行时/无障碍/自动化的同事，我希望画布上的 **按钮、文本、图片等叶节点** 在 DOM 中尽可能 **紧贴物料组件自身的根节点**，中间没有仅服务于「可选中、可拖拽」而存在的多余包裹层，这样 **审查元素、截图对比、与设计稿对齐** 时路径清晰，也减少样式继承与布局意外的来源。

**Why this priority**：若不做，用户感知的「页面结构」与导出/预览不一致，排障成本高；也是本条需求的直接动机。

**Independent Test**：在编辑器中选中一个典型叶节点（如按钮），在开发者工具中从组件可见根向上追溯，**不应**出现仅承担编辑器外壳、不含业务语义的多层 `div` 夹在画布内容区与组件根之间（与改造前对比可量化层数）。

**Acceptance Scenarios**:

1. **Given** 画布上有一个物料按钮叶节点，**When** 查看其 DOM 结构，**Then** 组件库输出的主可视根与画布中该节点的可视内容之间 **无** 仅为编辑器而设的嵌套包裹 `div` 链（允许保留项目统一要求的 **单一** 挂载点若合同另有规定，见 Assumptions）。
2. **Given** 该叶节点有内联样式与业务 props，**When** 渲染，**Then** 行为与改造前在视觉与交互上一致或更好，无「少包一层导致样式丢失」的回归。

---

### User Story 2 - 选中、悬停、缩放手柄不依赖包裹组件根 (Priority: P2)

作为编辑者，我仍需要 **选中高亮、拖拽移动、缩放控制点、悬停提示** 等能力；这些 **不得** 再依赖「在组件外包一层 div」才能实现，而应出现在 **与组件渲染子树并列或之上的覆盖层**（视觉上可贴合组件边缘）。

**Why this priority**：去掉包裹层后，若装饰层缺失，编辑不可用。

**Independent Test**：完成选中、拖拽、缩放、悬停全链路手工走查；装饰在视觉上仍对齐组件边界。

**Acceptance Scenarios**:

1. **Given** 叶节点被选中，**When** 查看页面，**Then** 可见选中框/手柄（或等价 affordance）且 **与组件外轮廓对齐**，不依赖组件根外新增结构层。
2. **Given** 鼠标悬停可编辑节点，**When** 移动指针，**Then** 悬停态仍可辨识。
3. **Given** 使用现有拖放与点击选中，**When** 操作，**Then** 与改造前成功率相当，无系统性无法选中或误选父级。

---

### User Story 3 - 容器类节点与多子树 (Priority: P3)

作为编辑者，我对 **容器**（含子节点的布局区域）仍需要能选中容器本身与子节点；在 **必须** 为子列表提供挂载点的场景下，允许 **最少量的结构节点**，但 **不应** 在「每个子叶节点」上重复套多层仅编辑器用的壳。

**Why this priority**：零包裹在「有子级」时往往不现实；本故事收敛边界、避免过度承诺。

**Independent Test**：内置容器 + 嵌套子组件场景下，子叶节点仍满足 US1 的层数预期；容器可选中、可配置。

**Acceptance Scenarios**:

1. **Given** 容器内嵌两个叶节点，**When** 分别选中子节点，**Then** 每个子节点仍满足 US1 的 DOM 清晰度预期（相对改造前减少冗余层）。
2. **Given** 选中容器，**When** 编辑，**Then** 容器级属性与选中态仍可用。

---

### Edge Cases

- **远程/异步物料**：组件根晚于外壳挂载时的闪烁、测量与 Portal 定位需可接受。
- **极小或零尺寸节点**：装饰层定位与可点区域仍可用。
- **与画布缩放、半屏展示**：Portal 层坐标需与当前画布变换一致，不出现错位。
- **无障碍与键盘遍历**：焦点顺序不因外置层而完全失控（最低限度不劣于当前）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 对于 **叶节点物料**（无子 schema 或合同认定的「非容器」节点），系统在画布中呈现其内容时，**不得** 仅为实现选中/拖拽而增加 **多级** 仅装饰性包裹节点；组件主输出与画布内容区之间的 DOM 深度 **不劣于** 本特性启动前所约定的目标（目标为 **减少** 冗余层，以走查与合同为准）。
- **FR-002**: 选中、悬停、缩放等 **编辑器装饰** 必须 **可在不包裹组件根的前提下** 正确显示与对齐；若采用子树外挂载，须保证与组件 **几何对齐** 随滚动、缩放更新。
- **FR-003**: 现有 **点击选中、拖拽移动、缩放、拖入组件、右键菜单** 等编辑能力 **保持可用**，无大面积失效。
- **FR-004**: **容器节点** 允许保留 **满足布局与递归渲染所必需** 的结构；在此前提下 **尽量** 不重复为每个子节点叠加无语义包裹。
- **FR-005**: **序列化 schema** 与 **运行时预览** 的语义 **不因 DOM 结构优化而改变**（仍仅以 schema 为真源）。

### Key Entities

- **叶节点**：无子或业务上不作为布局容器的 schema 节点；本条需求的主要对象。
- **编辑器装饰**：选中框、手柄、悬停描边等，非页面发布内容的一部分。
- **挂载层**：画布内用于承接 Portal 或覆盖层的稳定容器（由实现定义，规格只要求行为与对齐）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对 **不少于 3 个** 不同物料叶节点抽样，开发者工具中从 **组件主可视根** 向上到画布内容容器的 **仅编辑器用包裹 div 数量** **不多于** 改造前（理想为 **减少**），且评审签字确认「无多余壳」。
- **SC-002**: **10 次** 选中—拖拽—缩放循环中，**至少 9 次** 装饰与组件边缘 **肉眼对齐**，无固定方向的系统性偏移。
- **SC-003**: 原 **组件列表拖入、点击选中、属性联动** 主路径在回归清单中 **100%** 用例通过（与改造前基线一致）。
- **SC-004**: **2 名** 内部编辑者认为「结构更干净、与真实页面更一致」，且无人报告「无法选中或拖不动」的阻塞问题。

## Assumptions

- 「组件真实内容」以 **物料包导出的主渲染根** 为准；若第三方组件内部自有多个 DOM 节点，不在本条范围内削减。
- **Portal 或等价「子树外挂载」** 是实现 FR-002 的 **推荐方向**，但规格以 **用户可验证结果** 为准，不强制单一 API 名称。
- **容器** 因需挂载 `children`，允许 **最少一层** 与布局相关的结构；本特性 **优先交付叶节点** 的减包与装饰外置。
- **Preview / 发布页** 若与编辑器共用渲染路径，应 **一并受益或保持行为不变**；若路径分离，至少在编辑器内满足本规格。
