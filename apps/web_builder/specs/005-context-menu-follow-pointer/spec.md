# Feature Specification: 右键菜单锚定指针位置

**Feature Branch**: `005-context-menu-follow-pointer`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: User description: "点击鼠标右键时，出现的菜单栏跟随鼠标的位置"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 右键打开时菜单出现在指针附近 (Priority: P1)

作为编辑者，在画布或相关编辑区域**单击鼠标右键**打开上下文菜单时，我希望菜单在视觉上**以本次点击的指针位置为参照**出现，而不是固定在某个与点击无关的角落或屏幕中心，以便快速看到可选操作且符合常见桌面习惯。

**Why this priority**：定位正确是上下文菜单可用性的基础；错位会显著增加寻找菜单的认知负担。

**Independent Test**：仅验证「右键后菜单与点击点在视口内的相对关系合理」即可单独验收，无需改菜单项内容。

**Acceptance Scenarios**:

1. **Given** 用户在可触发上下文菜单的区域内，**When** 其按下并松开右键（或系统等效手势）打开菜单，**Then** 菜单的打开位置以该次指针坐标为锚点（例如菜单一角贴近指针常见落点），用户能直观感到「菜单在点击处出现」。
2. **Given** 菜单已打开，**When** 用户未选择任何项而移动指针，**Then** 菜单不随指针持续漂移（与常见桌面右键菜单一致）；关闭后再次右键则按新点击位置重新锚定。

---

### User Story 2 - 靠边与边界时的可见性 (Priority: P2)

作为编辑者，当我在**靠近视口边缘**或**可滚动区域边界**处右键时，我希望菜单仍**完整可见、可点**，系统通过自动调整展示方向或偏移避免菜单被裁切到屏幕外。

**Why this priority**：仅靠「固定锚在指针」在边界场景会失败；边界处理是完整体验的必要补充。

**Independent Test**：在视口四角及滚动容器边缘各做一次右键，确认菜单主体可见且主要操作项可点击。

**Acceptance Scenarios**:

1. **Given** 指针靠近视口右下侧且空间不足以在默认方向展开完整菜单，**When** 用户打开上下文菜单，**Then** 系统自动调整位置（如向反方向展开或向内偏移），使用户无需滚动页面即可看到并点击绝大部分菜单项。
2. **Given** 页面存在滚动，**When** 用户在滚动后的可见区域内右键，**Then** 锚定仍相对于**当前可见视口与指针**计算，不出现菜单完全落在不可见区域的情况。

---

### User Story 3 - 多类触发区域一致体验 (Priority: P3)

作为编辑者，凡是在编辑器内通过**同一套右键行为**打开的上下文菜单（若存在多处入口，如画布、节点、面板内列表等），我希望**定位规则一致**，避免一处跟随指针、另一处仍固定位置造成困惑。

**Why this priority**：一致性降低学习成本；可在核心区域先交付后再扩展到次要区域。

**Independent Test**：在规格或计划中列明的两类及以上触发区域各测一次，行为与 Story 1、2 一致。

**Acceptance Scenarios**:

1. **Given** 规格所覆盖的任意已列入范围的右键菜单入口，**When** 用户打开菜单，**Then** 均适用相同的「指针锚定 + 边界修正」规则（除非某入口在 Assumptions 中明确排除）。

---

### Edge Cases

- 视口尺寸很小或浏览器缩放比例很大时，菜单高度可能接近或超过可视高度：应保证至少可通过滚动或折叠策略访问主要项（具体策略在实现阶段选取，但不得出现完全不可达的菜单）。
- 若存在嵌套滚动容器，指针事件坐标与「视口」的关系需明确以**当前可见视口**为准，避免菜单落在被遮挡的滚动区域外。
- 使用键盘或辅助功能打开上下文菜单（若产品支持）时，锚点可回退到**当前焦点元素附近**或与系统约定一致；本规格以指针右键为主场景，其他方式在 Assumptions 中说明范围。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：在支持上下文菜单的交互中，当用户通过**指针右键**（或系统等效）打开菜单时，系统 MUST 使用**该次交互的指针位置**作为菜单定位的主要参考，使用户感知菜单在点击位置附近出现。
- **FR-002**：当按默认锚定方式会导致菜单**主要区域**落在当前可视区域外或不可点击时，系统 MUST 自动调整展示位置或偏移，使菜单在无需用户平移整个应用窗口的前提下即可操作。
- **FR-003**：菜单打开后，系统 MUST NOT 在用户仅移动指针（未关闭菜单）时持续将菜单跟随指针移动；再次打开时按新的点击位置重新计算。
- **FR-004**：在规格承诺覆盖的编辑器区域内，凡提供同类右键菜单的入口，其定位规则 MUST 一致（与 FR-001、FR-002 一致），除非在 Assumptions 中明确排除某入口。
- **FR-005**：系统 MUST 在因环境限制无法完成合理定位时，向用户提供可理解的降级表现（例如回退到安全默认位置），且不得静默失败为「无菜单」。

### Key Entities

- **指针打开位置**：用户触发右键菜单时，与事件关联的二维位置（相对当前可视区域），用作锚定参考。
- **可视区域（视口）**：用户当前可见的浏览器视口范围，用于边界检测与裁剪修正。
- **上下文菜单**：本次交互下展示的命令列表容器及其占位区域。

## Orange Editor / web_builder *(conditional)*

- **ARCH-001**：实现须符合 `.specify/memory/constitution.md`（扩展/插槽、状态分层、不在核心随意挂载无关逻辑）；若菜单由扩展贡献，应通过既有扩展与注册机制接入，避免重复多套右键实现。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：在标准桌面浏览器环境下，抽样至少 **5** 次不同屏幕区域内的右键打开，**100%** 情况下菜单主体出现在指针所在象限的合理邻域内（走查或录屏量测，允许为实现所做的边界翻转）。
- **SC-002**：在视口**四边及四角**各至少 **1** 次用例中，菜单均**完整可见**且首屏可点击项不少于 **80%**（若菜单项过多，以常用项是否可达为准，并在验收记录中注明菜单规模）。
- **SC-003**：**90%** 以上参与走查的成员认为「菜单出现位置符合对右键菜单的预期」（简短短问卷或举手统计即可）。
- **SC-004**：在计划列出的 **2** 类及以上触发区域中，走查未发现「一类跟随指针、一类完全固定角落」的**无文档化**不一致（已文档排除的入口除外）。

## Assumptions

- 「跟随鼠标的位置」指**打开瞬间**以指针位置为锚，**而非**菜单显示期间持续跟踪指针移动；与主流操作系统右键菜单行为一致。
- 主目标平台为**桌面端**带指针设备；触控长按菜单若存在，可作为后续迭代对齐，不阻塞本规格 P1/P2 验收。
- 具体哪些 UI 表面纳入「必须一致」范围（Story 3）将在 **plan** 阶段列出清单；未列入的表面不在 SC-004 强制范围内。
- 键盘打开菜单的锚点规则：若有产品既有约定则遵循，否则允许与指针路径略有差异，但须在实现说明中写明。
