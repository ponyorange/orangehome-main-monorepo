# Feature Specification: 设置内主题切换与右上角用户展示

**Feature Branch**: `002-settings-theme-user`  
**Created**: 2026-03-27  
**Status**: Draft  
**Input**: User description: "主题切换放到左上角设置按钮里，点击设置按钮，展示一个下拉菜单，目前这个下来菜单只有一个选项，就是「切换主题」。然后右上角加上展示用户的头像和昵称。如果没有头像url，就用一个默认的高级头像代替。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 通过左上角设置打开主题切换 (Priority: P1)

用户在编辑器界面需要使用主题切换能力时，点击**左上角**的**设置**入口，出现下拉菜单；菜单中当前仅有**一项**：「切换主题」。选择该项后，用户能够完成与产品预期一致的主题切换（明暗/套色等现有能力保持可用，仅入口迁移至设置菜单）。

**Why this priority**: 主题切换是明确交付项，且设置菜单的结构由此建立，其它后续菜单项可在此基础上扩展。

**Independent Test**: 在未依赖右上角用户信息的前提下，仅通过设置按钮与下拉菜单验证「切换主题」可达且生效。

**Acceptance Scenarios**:

1. **Given** 用户处于编辑器主界面，**When** 点击左上角设置按钮，**Then** 出现下拉菜单且可见「切换主题」选项。
2. **Given** 下拉菜单已打开，**When** 用户选择「切换主题」，**Then** 系统提供或进入与当前产品一致的主题切换方式，且切换后界面主题状态正确更新。
3. **Given** 下拉菜单已打开，**When** 用户点击菜单外区域或按合理方式关闭菜单，**Then** 菜单关闭且不误触其它危险操作。

---

### User Story 2 - 右上角展示头像与昵称 (Priority: P2)

用户在编辑器主界面**右上角**应持续看到**当前登录用户**的**头像**与**昵称**（或等价显示名），便于确认身份。当用户资料中**没有头像 URL** 时，展示**统一的默认头像**，风格需简洁、有品质感（「高级」观感），避免简陋占位图。

**Why this priority**: 与设置菜单可并行交付；依赖已登录态与用户信息来源，但不应阻塞设置菜单本身的可用性验收。

**Independent Test**: 在固定主题切换入口不变的前提下，仅验证右上角展示与缺省头像行为。

**Acceptance Scenarios**:

1. **Given** 用户已登录且资料含头像 URL 与昵称，**When** 进入编辑器主界面，**Then** 右上角展示该头像与昵称。
2. **Given** 用户已登录但无头像 URL，**When** 进入编辑器主界面，**Then** 右上角展示默认高品质头像与昵称（或可用显示名）。
3. **Given** 用户已登录但昵称缺失，**When** 进入编辑器主界面，**Then** 右上角仍展示头像（或默认头像）并对名称使用合理回退（如邮箱前缀或其它已有字段），不出现空白无说明区域。

### Edge Cases

- 头像 URL 无效或加载失败：应有与「无 URL」一致的默认头像表现或等价降级，不出现长期裂图占坑。
- 昵称过长：在右上角区域内**截断或省略**展示，不撑破顶栏布局。
- 窄屏或缩放：设置按钮、下拉菜单与右上角用户信息在常见桌面宽度下仍**可点、可读**；极端窄屏下以不重叠、不遮挡关键编辑区为底线。
- 未登录或会话失效：不应与「编辑器内已登录主界面」场景矛盾；若仅登录后进入编辑器，则本规格默认验收在**已登录**前提下进行。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在编辑器主界面**左上角**提供**设置**按钮（或图标按钮），语义清晰为「设置」。
- **FR-002**: 用户点击设置按钮后，系统 MUST 展示**下拉菜单**，且当前菜单中**至少包含**一项文案为「切换主题」的选项。
- **FR-003**: 用户选择「切换主题」后，系统 MUST 允许用户完成主题切换，且切换结果与全局主题状态一致（与产品现有主题能力对齐，不因搬家而丢失能力）。
- **FR-004**: 系统 MUST 在编辑器主界面**右上角**展示当前用户的**头像**与**昵称**（或等价显示名）。
- **FR-005**: 当用户无头像 URL 或头像不可用时，系统 MUST 展示**默认头像**，视觉风格为简洁、有品质感的统一占位，而非低质随意图标。
- **FR-006**: 下拉菜单 MUST 可通过常见交互关闭（如点击外部、再次点击设置或 Esc，以实现为准），且关闭后状态可再次打开。

## Orange Editor / web_builder *(conditional)*

- **ARCH-001**: Implementation MUST follow `.specify/memory/constitution.md` (extensions/slots, schema source of truth, DI, stores, `src/data/api`, public exports).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在不少于 3 名试用者参与的走查中，所有人能在**首次尝试或一次纠正内**找到设置并打开含「切换主题」的下拉菜单。
- **SC-002**: 在已登录编辑器主界面场景下，**100%** 的抽样检查中右上角同时出现可辨识的头像区域与名称区域（名称可为昵称或合规回退）。
- **SC-003**: 对无头像账号的抽样账号，**100%** 显示默认头像且无裂图；验收人员主观评价默认头像「不低于现有产品其它占位图品质」。
- **SC-004**: 主题切换完成后，**一次刷新或同会话内导航**后主题状态保持一致，无随机回退。

## Assumptions

- 验收场景默认用户**已登录**并已进入**编辑器主界面**（与当前 Builder 登录后编辑流程一致）。
- 「切换主题」所调用的具体主题能力（如面板、弹层或侧栏）可与现状一致，本规格只要求**入口**迁至设置下拉菜单内链接触达。
- 用户信息（头像 URL、昵称、邮箱等）来自现有登录/用户接口或前端已有数据源，不强制本特性新增后端字段。
- 「左上角 / 右上角」相对**编辑器顶栏**区域；若顶栏由插槽/扩展组成，实现仍须满足视觉位置约定。
