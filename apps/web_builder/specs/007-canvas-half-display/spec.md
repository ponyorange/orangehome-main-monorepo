# Feature Specification: 编辑区画布半屏展示（逻辑仍 750 幅面）

**Feature Branch**: `007-canvas-half-display`  
**Created**: 2026-03-29  
**Status**: Draft  
**Input**: User description: "现在画布区是750*1334，展示的有点大，画布区缩小一半展示把，但实际上他们还是750宽度的，我开发组件也是按750来开发的。只是中间画布区展示效果上按375的效果来展示，也就是缩小一半。可以理解为右上角的缩放缩放了50%，但展示还是100%"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 中间编辑区视觉上更小，设计坐标不变 (Priority: P1)

作为页面搭建编辑者，我希望中间画布在屏幕上占用的面积约为现在的一半（宽高各约一半），这样一屏能留出更多周边空间、减少「画布太大」的感觉；同时我放置组件、看属性里的位置与宽高时，仍然按 **750 宽** 的页面来理解和编排，与真实逻辑幅面一致。

**Why this priority**：若不先满足「视觉缩小 + 逻辑不变」，要么画布仍过大，要么坐标与心智不一致，直接影响日常搭建效率与正确性。

**Independent Test**：仅开放此能力时，用同一套页面数据对比改造前后：属性/坐标仍以 750 为基准；肉眼可见画布区域线性尺寸约为改造前的一半（在相同缩放档位下）。

**Acceptance Scenarios**:

1. **Given** 使用默认编辑状态，**When** 查看中间白底画布区域，**Then** 其外轮廓的宽高约为改造前同条件下的约一半（线性比例约 50%）。
2. **Given** 画布上已有占满「逻辑宽度」的组件，**When** 查看该组件在属性中的宽度或与 750 相关的描述，**Then** 仍与 750 幅面一致，不因视觉缩小而变成 375 的语义。
3. **Given** 在画布上拖拽、对齐、选择，**When** 完成操作，**Then** 落点与选中行为与逻辑坐标系一致，不出现因「只看屏幕像素」而产生的双倍或减半误差。

---

### User Story 2 - 缩放控件的「100%」含义不被误标为 50% (Priority: P2)

作为编辑者，我希望右上角（或等价位置）的缩放显示仍表达「相对设计幅面的缩放」含义；视觉上的「整块画布变小」不应单独表现为「我已经只有 50% 缩放」——即：**默认下界面仍显示 100%（或你们产品既有的基准文案），而不是把半屏展示误当成把缩放条变成 50%。**

**Why this priority**：若缩放数字与心智冲突，用户会怀疑导出、预览与编辑是否一致，增加支持与误操作成本。

**Independent Test**：默认进入编辑器，缩放读数与改造前在「未额外缩放」档位下的文案一致；同时画布视觉尺寸已为半屏效果。

**Acceptance Scenarios**:

1. **Given** 首次打开编辑器（默认缩放档位），**When** 查看缩放指示，**Then** 不出现「仅因半屏展示而被动显示为 50%」的误导（应与产品对「100% = 设计基准缩放」的定义一致）。
2. **Given** 用户将缩放从 100% 调到 200%，**When** 观察画布，**Then** 相对默认半屏基底的放大关系可预期、可重复，且读数与实际操作一致。

---

### User Story 3 - 与标尺、参考线等辅助信息一致 (Priority: P3)

作为编辑者，我在使用标尺、网格或对齐参考时，读到的刻度与逻辑幅面应对得上：半屏只是「怎么看」，不是「量出来变成另一套数」。

**Why this priority**：辅助线与数字不一致会导致对齐困难和信任问题。

**Independent Test**：在默认状态下，标尺末端与画布逻辑边界在语义上仍对应 750×1334（或与产品当前合同一致的设计幅面），不因视觉缩小而显示成 375 作为「整页宽度」。

**Acceptance Scenarios**:

1. **Given** 显示横向标尺，**When** 看到与整页逻辑宽度对应的末端刻度，**Then** 与 750 逻辑幅面一致（而非把整页标成 375）。
2. **Given** 使用缩放，**When** 改变缩放档位，**Then** 标尺与画布的相对关系保持清晰、可读，不出现刻度与可编辑区域脱节。

---

### Edge Cases

- 极小或极窄的编辑器窗口：半屏展示后画布是否仍可滚动、是否出现裁切，应保持与现有编辑器一致的可用性底线。
- 高缩放（如 200%、300%）：视觉与滚动范围应仍可用，不出现无法拖回画布的情况。
- 与「预览 / 真机预览」的关系：若产品另有独立预览窗口，其设备框尺寸策略是否与本特性解耦，应在实现计划中明确边界（本规格默认：**仅约束中心编辑画布区域**的展示与逻辑坐标关系，不强制改变独立预览产品的既有约定）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 保持页面/schema 及属性面板所表达的画布逻辑幅面为 **750×1334**（或与当前产品合同一致的编辑幅面），组件开发与编排仍按该幅面理解。
- **FR-002**: 系统 MUST 使中心编辑画布区域在默认情况下的**屏幕上线性尺寸**约为改造前的 **50%**（宽高各约一半），从而接近「以前 375 宽那类视觉占地」的体验。
- **FR-003**: 系统 MUST 区分「视觉展示比例」与「缩放控件所表示的设计缩放」：默认状态下缩放读数 MUST 符合产品对 **100%（或等价基准档）** 的定义，不得仅因半屏展示而向用户呈现「已是 50% 缩放」的误导性标签（除非产品明确将基准档重定义为其他文案并在全链路一致）。
- **FR-004**: 系统 MUST 保证拖拽、选择、对齐、吸附等行为所依据的坐标与逻辑幅面一致，不因视觉缩小而产生坐标系错位。
- **FR-005**: 系统 MUST 使标尺、网格等与画布逻辑边界相关的辅助信息在语义上与逻辑幅面一致（整页宽度仍对应 750 而非 375）。
- **FR-006**: 用户 MUST 仍可通过现有缩放控件放大或缩小编辑视图，且读数与视觉变化的关系清晰、可预测。

### Key Entities

- **逻辑画布（设计幅面）**：页面在数据与属性中的宽与高（750×1334）；拖拽与布局的权威坐标系。
- **中心编辑区视觉展示**：用户在中间区域看到的画布像素占地；可通过与逻辑幅面解耦的展示比例调整，而不改写逻辑幅面。
- **缩放档位（用户可见）**：用户理解的「放大/缩小页面」的数值或标签，应对应设计空间缩放语义，而非单独吞掉半屏展示带来的比例。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在相同默认缩放档位下，不少于 **3 名**内部试用者能一致描述「画布在屏幕上变小了约一半」，且无人认为「页面逻辑宽度从 750 变成了 375」。
- **SC-002**: 对同一测试页面，改造前后属性中「全宽」类组件的宽度数值或等价描述 **100%** 一致（允许显示格式差异，不允许语义减半）。
- **SC-003**: 默认进入编辑器时，缩放相关文案/读数与改造前默认档 **一致**（不出现仅因半屏展示而被动显示为 50% 的单一归因），抽样 **5 次**加载均成立。
- **SC-004**: 在标尺开启时，试用者能在 **2 分钟内**确认「末端刻度与整页逻辑宽度」的关系与预期一致（与 750 幅面对齐），无「整页被标成 375」的报告。

## Assumptions

- 编辑幅面合同仍为 **750×1334**，与本特性之前已采纳的规格一致；本特性不改变该合同，只改变中心区的**视觉呈现比例**与**缩放读数语义**的划分方式。
- 「缩小一半」指**线性尺寸**各约 50%；面积约为原来的 25%。
- 独立「预览」或对外发布视图若有单独尺寸策略，默认不在本规格内强制统一，除非产品后续明确要求。
- 用户主要在桌面宽屏上使用编辑器；极端窗口尺寸下的体验以不低于当前编辑器可用性为底线。
