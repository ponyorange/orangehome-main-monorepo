# Feature Specification: 拖动组件到容器上方添加组件、计算鼠标落点、单击添加组件到选中容器

**Feature Branch**: `[013-drag-drop-component-addition]`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: "拖动组件到画布区时，如果拖动了组件到容器上方，就把组件添加到该容器中。计算组件落点位置，组件要落在鼠标位置。除了拖动添加组件外，单击组件时，可以将组件添加到当前选中的容器组件中，如果没有选中的容器，就添加到根容器中。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 拖动组件到容器上方自动添加 (Priority: P1)

作为编辑器用户，我希望在拖动组件到画布区时，如果拖放到了某个容器组件的上方，则自动将该组件添加到该容器中，以便更直观地组织页面结构。

**Why this priority**: 这是最核心的交互优化，解决用户需要将组件放入容器的常见场景，大幅提升编排效率。

**Independent Test**: 可以通过从物料面板拖动任意组件到画布上的容器（如div、卡片等）进行测试。拖动时容器应有视觉反馈，释放后组件应出现在容器内正确的位置。

**Acceptance Scenarios**:

1. **Given** 画布上有一个容器组件（如div、卡片），**When** 用户从物料面板拖动一个组件到该容器上方，**Then** 容器显示视觉反馈（如边框高亮），释放后组件被添加到容器内
2. **Given** 画布上有多个嵌套容器，**When** 用户拖动组件到最内层容器上方，**Then** 组件被添加到最内层容器而非外层容器
3. **Given** 用户拖动组件但没有拖放到任何容器上方，**Then** 组件被添加到根画布（与现有行为一致）

---

### User Story 2 - 组件精确落在鼠标位置 (Priority: P1)

作为编辑器用户，我希望组件被添加时精确落在鼠标释放的位置上，而不是默认的左上角或其他固定位置，以便更精确地控制组件布局。

**Why this priority**: 精确的落点控制是直观编辑的基础需求，与容器添加功能相辅相成，共同构成完整的拖拽体验。

**Independent Test**: 可以拖动组件到画布的任意位置进行测试。释放后组件的左上角应对齐鼠标指针位置，或相对于容器原点计算正确坐标。

**Acceptance Scenarios**:

1. **Given** 用户从物料面板拖动组件，**When** 在画布任意位置释放，**Then** 组件被添加在鼠标指针所在位置（考虑画布缩放因子）
2. **Given** 用户将组件拖入容器内，**When** 在容器内部某位置释放，**Then** 组件坐标相对于容器原点计算，落点在鼠标相对于容器的位置
3. **Given** 画布有缩放（如放大/缩小），**When** 用户释放组件，**Then** 组件被添加到正确的逻辑坐标位置（考虑缩放因子转换）

---

### User Story 3 - 单击添加组件到选中容器 (Priority: P2)

作为编辑器用户，我希望除了拖动外，还可以通过单击物料面板中的组件来快速添加组件，组件应被添加到当前选中的容器中（如果没有选中容器则添加到根画布），以便更快速地构建页面。

**Why this priority**: 单击添加是拖动操作的便捷替代，特别适合快速连续添加多个组件的场景。优先级略低于拖拽功能，因为拖拽是更直观的交互方式。

**Independent Test**: 可以先选中画布上的某个容器，然后单击物料面板中的任意组件，组件应被添加到选中的容器内。如果没有选中任何容器，组件应被添加到根画布。

**Acceptance Scenarios**:

1. **Given** 用户在画布上选中了一个容器组件，**When** 用户单击物料面板中的某个组件，**Then** 该组件被添加到选中的容器内，位置默认为容器中心或左上角
2. **Given** 用户没有选中任何组件（或选中了非容器组件），**When** 用户单击物料面板中的组件，**Then** 该组件被添加到根画布
3. **Given** 用户选中了多个组件（多选状态），**When** 用户单击物料面板中的组件，**Then** 如果选中的第一个组件是容器，则添加到该容器；否则添加到根画布

---

### Edge Cases

1. **What happens when** 拖动组件到嵌套容器的边界区域（同时处于多个容器的上方）？
   - 应使用命中测试找到最内层（最具体）的容器
2. **How does system handle** 拖动到非容器组件（如按钮、文本）上方？
   - 应忽略非容器组件，继续向上查找容器或添加到根画布
3. **What happens when** 容器的子元素已经填满了容器的可见区域？
   - 组件仍应被添加到容器内，可能需要滚动或溢出显示
4. **How does system handle** 鼠标位置超出画布边界？
   - 应限制在画布有效区域内，或取消添加操作
5. **What happens when** 单击添加时目标容器已经包含大量子组件？
   - 新组件应添加到合适位置（如列表末尾），不应覆盖现有组件

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持在拖动组件到画布时识别下方的容器组件
- **FR-002**: 系统必须在拖动过程中为潜在目标容器提供视觉反馈（如边框高亮、背景变化）
- **FR-003**: 系统必须将组件添加到最内层的容器（当多个容器嵌套时）
- **FR-004**: 系统必须计算鼠标释放位置相对于目标容器原点的坐标，组件左上角对齐鼠标位置
- **FR-005**: 系统必须考虑画布缩放因子将视觉坐标转换为逻辑坐标
- **FR-006**: 系统必须支持单击物料面板组件来添加组件到当前选中的容器
- **FR-007**: 系统在没有选中容器时必须将组件添加到根画布
- **FR-008**: 系统必须区分容器组件和非容器组件，仅向容器添加子组件
- **FR-009**: 系统必须在单击添加时为组件提供合理的默认位置（相对于容器的左上角偏移20x20像素，如果该位置已被占用则递增偏移）

### Key Entities *(include if feature involves data)*

- **容器组件（Container）**: 可以包含子组件的组件类型，具有children属性，如div、卡片、布局容器等
- **非容器组件（Leaf Component）**: 不能包含子组件的基础组件，如按钮、文本、图片等
- **命中测试（Hit Testing）**: 根据鼠标位置确定当前位于哪个组件上方的算法
- **视觉坐标（Visual Coordinate）**: 屏幕上看到的像素坐标，受画布缩放影响
- **逻辑坐标（Logical Coordinate）**: 存储在schema中的实际坐标，与缩放无关

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在3秒内成功将组件拖入目标容器，成功率达到95%
- **SC-002**: 组件落点位置与鼠标位置的偏差不超过5像素（考虑缩放因子后）
- **SC-003**: 单击添加组件到选中容器的操作成功率达到100%（当存在有效选中容器时）
- **SC-004**: 新功能对用户操作的平均响应时间不超过100毫秒

## Assumptions

- 容器组件可以通过类型或属性标识（如 `isContainer: true` 或特定的组件类型列表）
- 现有物料面板和画布拖拽基础设施可以扩展以支持新功能
- 画布缩放因子可以通过现有store获取
- 当前选中组件的状态可以通过现有selection store获取
- 坐标计算需要考虑绝对定位和相对定位两种场景
