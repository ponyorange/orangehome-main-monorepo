# Feature Specification: 右侧配置面板（参考组件 Inspector 稿）

**Feature Branch**: `003-right-panel-inspector-ui`  
**Created**: 2026-03-27  
**Status**: Draft  
**Input**: User description: "@component-inspector-panel1.0.html 参考这个ui，重做编辑器右侧配置面板。"  
**Visual / IA Reference**: `refer_ui/component-inspector-panel1.0.html`（布局、分区、信息层级与视觉气质的权威参考；实现方式不限定于该静态稿的技术栈）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在统一「配置」视图中完成属性与样式编辑 (Priority: P1)

作为页面编辑者，在选中画布上的组件后，我需要在右侧面板的一个「配置」区域内，按清晰分组查看并修改**业务属性**与**外观样式**，表单布局与参考稿一致（右侧控件、左侧标签对齐、分组标题醒目），以便快速完成常见调整而无需在多个割裂的视图间来回切换。

**Why this priority**: 这是编辑器的核心工作闭环；面板结构直接决定可发现性与操作效率。

**Independent Test**: 仅交付「配置」分区与分组表单壳层及与现有编辑能力的绑定即可独立验收（可不先重做「信息」分区）。

**Acceptance Scenarios**:

1. **Given** 已选中一个可编辑组件，**When** 打开右侧面板并处于「配置」分区，**Then** 可见与参考稿同构的**头部标题区**（组件显示名 + 一句用途说明）以及**分段式顶栏**中的「配置」为当前选中态。
2. **Given** 同上，**When** 浏览「配置」内容，**Then** 至少包含**属性配置**与**样式配置**两个分组；每组内为「标签 + 控件」栅格布局，标签右对齐，控件占剩余宽度，与参考稿视觉层级一致（分组小标题、强调色、间距与圆角气质一致）。
3. **Given** 用户修改某一表单项，**When** 变更生效，**Then** 画布或 schema 中与该字段对应的展示与持久化行为与编辑器现有规则一致（不因换肤而破坏撤销/重做或数据源一致性）。

---

### User Story 2 - 在「信息」分区查看标识与 Schema (Priority: P2)

作为编辑者或排错人员，我需要在「信息」分区只读查看组件的稳定标识（如实例 ID、类型标识），并在同区以可读方式浏览**当前组件相关的 Schema 片段**（或等价结构化说明），便于沟通、复制或对照协议，而不必在「配置」里混杂只读长文本。

**Why this priority**: 降低支持与协作成本；与参考稿「配置 / 信息」二段式一致。

**Independent Test**: 在「配置」仍为旧版时也可先只做「信息」分区与 Tab 切换，但完整价值在两项联调后体现。

**Acceptance Scenarios**:

1. **Given** 已选中组件，**When** 切换到「信息」分区，**Then** 展示键值行列表：至少包含**组件 ID**与**组件 Type**（或产品约定的类型展示名）；类型可用醒目的徽章样式呈现，与参考稿一致。
2. **Given** 同上，**When** 查看 Schema 区域，**Then** 在独立 titled 区块内展示格式化的结构化文本（可滚动、等宽字体、浅色底与边框与参考稿气质一致），内容与当前选中组件的 schema 定义一致。
3. **Given** 未选中任何组件或选中不可展示项，**When** 打开「信息」分区，**Then** 显示明确的空态或提示文案，不出现误导性空白或陈旧数据。

---

### User Story 3 - 面板整体体验与可访问性 (Priority: P3)

作为长时间使用者，我需要右侧面板在固定宽度下**可滚动**、**焦点与键盘可操作**（顶部分段、表单控件、滚动区域），并在窄视口下不出现横向溢出或不可达控件。

**Why this priority**: 影响满意度与合规基础，次于核心编辑与信息架构。

**Independent Test**: 用键盘与缩小窗口走查一遍即可完成验收。

**Acceptance Scenarios**:

1. **Given** 配置项超过可视高度，**When** 用户在面板内滚动，**Then** 仅内容区滚动，头部与分段控件保持可见或按参考稿约定固定，且滚动条样式与参考稿一致（细轨道、与主题色协调）。
2. **Given** 用户使用键盘，**When** 在分段控件上切换，**Then** `aria-selected` / 焦点环与参考稿的可聚焦行为一致，且屏幕阅读器能识别分区与面板标题。

---

### Edge Cases

- 选中项在编辑过程中被删除或替换：面板应同步清空或切换到合法空态，不展示过期 ID。
- Schema 体积极大：信息区应可滚动且保持性能可接受（例如明显卡顿不应超过产品基线）。
- 某组件无样式或无部分属性：对应分组隐藏或显示「不适用」说明，避免空白误导。
- 主题切换（亮/暗或品牌色）：面板应保持与全局编辑器主题协调；若参考稿为浅橙系，产品可约定「在默认主题下与参考稿对齐，暗色主题为语义等价适配」。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 右侧配置面板 MUST 采用与参考稿一致的**整体壳层**：圆角卡片、外阴影、顶栏标题 + 副标题、左侧竖向强调条（或等价视觉锚点），宽度与参考稿同量级（约 350px 可视目标，允许随产品栅格微调）。
- **FR-002**: 面板 MUST 提供**两个顶部分段**：「配置」与「信息」，互斥选中；默认选中「配置」；切换后内容区立即切换且无数据串区。
- **FR-003**: 「配置」分区 MUST 包含**属性配置**与**样式配置**（及未来由 schema 驱动的其他分组），每组带分组标题样式（小写宽字母间距或产品等价样式）与表单项栅格；控件类型须覆盖当前编辑器已支持的属性/样式字段类型（数值、下拉、文本、颜色等），呈现方式与参考稿一致（如颜色行：色块预览 + 文本输入）。
- **FR-004**: 「信息」分区 MUST 展示**组件实例 ID**、**类型标识**（徽章），以及**可滚动的 Schema 预览区**，内容对应当前选中节点。
- **FR-005**: 所有编辑操作 MUST 继续遵守编辑器既有数据源与撤销语义；本特性仅改变**呈现与信息架构**，不引入第二套文档真相。
- **FR-006**: 面板 MUST 在内容溢出时提供**垂直滚动**，且头部/分段区域行为符合参考稿（通常固定于顶部）。
- **FR-007**: 面板 MUST 满足基础可访问性：分段控件具备正确的 `role`/`aria-selected`（或等价模式）、表单控件具备可关联的 label。

### Key Entities

- **选中节点**：当前在画布上选中的组件实例；驱动面板标题、表单字段与信息区内容。
- **配置分组**：属性配置、样式配置等逻辑分组；映射 schema 字段集合。
- **信息快照**：ID、类型、Schema 文本（或格式化结果）；只读、随选中节点更新。

## Orange Editor / web_builder *(conditional)*

- **ARCH-001**: Implementation MUST follow `.specify/memory/constitution.md`（扩展与插槽、schema 为真相、分层状态、DI 边界、`src/data/api` 约定、包导出契约）。
- **ARCH-002**: 右侧面板贡献方式 MUST 仍通过既有扩展与 `right-panel` 相关插槽（或计划阶段明确的等价注册点），避免在核心中硬编码孤立挂载。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 目标用户在**无培训**情况下，能在 **60 秒内**完成：选中组件 → 打开「配置」→ 找到「样式配置」→ 完成一项可见样式修改并能在画布上确认生效。
- **SC-002**: 在**同一选中会话**内切换「配置」与「信息」并返回，**100%** 情况下表单值与信息区内容与当前选中节点一致（抽样至少 5 类不同组件类型验收）。
- **SC-003**: 可用性走查中，**至少 80%** 参与者认为右侧面板「分区清晰、与参考稿一致或更好」，且**无人**将「信息」与「配置」的主要用途混淆。
- **SC-004**: 在标准测试页面上，面板内滚动与 Tab 切换交互在**主观上流畅**（无明显掉帧或超过 **300ms** 的可见阻塞）于目标设备基线。

## Assumptions

- 参考 HTML 文件为**产品与视觉结构的验收基准**，允许在工程上换用设计体系组件，但**分区命名、分组结构、关键控件组合（如颜色行）**须一致或经产品书面确认偏差。
- 当前编辑器已具备属性/样式/schema 的数据与更新通路；本特性以**重做 UI 与 IA**为主，不强制在本次扩展 schema 能力边界。
- 「配置」中的属性与样式字段集合以**现有 schema** 为准；参考稿中的示例字段（如可见、布局枚举等）若与真实 schema 不一致，以实现数据为准、布局遵循参考稿。
- 暗色或高对比主题若与浅橙参考稿冲突，采用**语义色映射**即可，不要求像素级同色。
